Code Explorer: main.py

Interactive learning dashboard for the Pygame moving squares simulation.

Architecture Overview

Runtime Layers

  • Setup: constants + init_game() + create_squares()
  • Loop control: run_loop() and handle_events()
  • Simulation: update_squares(), flee(), calculator()
  • Rendering: draw_scene()

Data Model

  • Each square is a dict: x, y, size, vx, vy, color
  • Main state is squares: list[dict]
  • One simulation step: move -> bounce -> compute flee vector
  • Behavior coupling: flee logic reads a static per-frame snapshot

Function-by-Function Notes

init_game
Bootstraps Pygame display and clock. Good separation of setup concerns.
speed_from_size
Linear mapping from size to speed. Includes defensive fallback when range is invalid.
create_squares
Random spawn position, color, angle-based velocity. Produces diverse initial conditions.
handle_events
Clean quit path: window close, Escape, and Q key.
calculator
Finds nearest larger threat in flee radius and returns away-vector direction.
flee
Only smaller squares flee; keeps original speed magnitude while changing heading.
update_squares
Moves squares, applies boundary bounce, then computes behavior phase.
draw_scene
Draws background, FPS HUD, and all rectangles, then flips display.
run_loop
Main scheduler. Improvement candidate: move font creation outside while-loop.

Pattern and Quality Analysis

Game Loop Pattern: strong Type Safety: medium Data Structure Clarity: medium Behavior Decomposition: good
  • Good modular decomposition into setup, update, and render phases.
  • Using dict for squares is flexible but fragile for typos and missing keys.
  • calculator() reads tuple snapshots while other code uses dicts; mixed representations increase cognitive load.
  • flee() has an empty branch (pass) that can be simplified by early return.

Complexity and Performance Notes

  • Per frame complexity is roughly O(n^2) due to threat checks for each square.
  • At SQUARE_COUNT = 50, this is usually fine; at larger counts, neighbor partitioning can help.
  • Font allocation is inside the hot loop; cache with one-time pygame.font.SysFont(None, 28).
  • Distance uses sqrt; compare squared distance to avoid repeated square roots in tight loops.

Improvement Checklist

  • Replace square dicts with a typed @dataclass (clear fields, safer refactors).
  • Add return type hints for calculator() and flee().
  • Unify square representation (all dict or all dataclass/tuple, not mixed).
  • Move HUD font creation outside loop to reduce per-frame allocations.
  • Add tiny unit tests for speed_from_size() and flee direction edge cases.
